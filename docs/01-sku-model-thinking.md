# 从数组套数组到标准 SKU 模型：一次规格设计的认知升级

> 关键词：SKU、规格设计、数据结构、状态建模

---

## 前言

在电商类 App 中，SKU（Stock Keeping Unit，最小库存单元）几乎是一个**必做功能**。

但它同时也是一个**极容易被写乱**的功能。

很多开发者在第一次实现 SKU 时，都会有类似经历：

* 一开始写得很顺手
* 规格能选、价格能变、库存也能判断
* 但需求一扩展，代码就开始失控

回头一看，问题往往不在算法，而在**一开始的数据结构设计**。

这篇文章想做的事情只有一件：

> **帮你从“数组套数组”的直觉实现，升级到一套标准、可扩展的 SKU 认知模型。**

---

## 一、为什么我们会自然写出「数组套数组」

### 1. 最常见的直觉建模方式

以一个典型商品为例：

* 颜色：红色 / 蓝色
* 尺码：M / L
* 套餐：A / B

很多人第一次写 SKU，会顺着「用户的选择路径」去建模：

```
红色
 ├── M
 │    ├── 套餐A
 │    └── 套餐B
 └── L
      ├── 套餐A

蓝色
 └── M
      └── 套餐A
```

映射到代码里，通常就是：

* 第一层：颜色数组
* 每个颜色下面：尺码数组
* 每个尺码下面：套餐数组
* 最底层才是真正的商品数据

### 2. 这种方式为什么“看起来很合理”

这种结构在初期有明显优点：

* 数据结构和 UI 层级高度一致
* 选中上一级，自然只能看到下一级
* 不需要额外的匹配算法

在 Demo 阶段，它**几乎不会出问题**。

问题通常出现在：

> **你开始把它当成一个“长期维护的功能”时。**

---

## 二、数组套数组的核心问题

### 问题 1：数据冗余和一致性灾难

在嵌套结构中，同一个 SKU 信息会被拆散在不同路径下：

* 价格
* 库存
* SKU id

一旦某个 SKU 的库存变化：

* 你需要在多个节点同步修改
* 稍不注意就会出现数据不一致

这类问题**不是写代码能避免的，而是结构决定的**。

---

### 问题 2：规格维度被“写死”

嵌套结构本质上隐含了一个前提：

> 规格是有固定顺序的。

例如：

```
颜色 → 尺码 → 套餐
```

一旦需求变化：

* 新增一个「版本」规格
* 或者调整规格展示顺序

你会发现：

> **整个数据结构和代码逻辑都需要推倒重来。**

---

### 问题 3：无法清晰表达「不可选」与「无库存」

这是数组套数组最致命的问题之一。

在嵌套结构中：

* 看不到的选项 ≠ 不存在
* 更无法区分：

    * 不存在该组合
    * 存在但库存为 0

结果就是：

* UI 只能“裁剪数据”来表现状态
* 业务语义完全丢失

---

### 问题 4：业务逻辑被迫写进 UI

由于数据已经是“被裁剪过的结果”：

* Adapter 只能基于当前层级判断
* Fragment 里开始出现大量 if / else

最终的结果是：

> **SKU 规则分散在各个 UI 层，既难维护，也无法复用。**

---

### 小结一句

> **数组套数组，是为了“当前页面好写”，而不是为了“业务长期正确”。**

---

## 三、重新认识 SKU：规格 ≠ SKU

要走出嵌套结构，关键是转变一个认知：

> **规格（Spec）和 SKU 是两种完全不同的东西。**

---

### 1. 规格列表：只负责“页面展示”

页面上展示的，是“规格维度”，而不是 SKU。

例如：

* 颜色：红色 / 蓝色
* 尺码：M / L
* 套餐：A / B

对应的数据结构应该是**扁平、独立的**：

```json
{
  "specs": [
    {
      "specId": "color",
      "name": "颜色",
      "values": [
        { "id": "red", "name": "红色" },
        { "id": "blue", "name": "蓝色" }
      ]
    },
    {
      "specId": "size",
      "name": "尺码",
      "values": [
        { "id": "m", "name": "M" },
        { "id": "l", "name": "L" }
      ]
    },
    {
      "specId": "package",
      "name": "套餐",
      "values": [
        { "id": "a", "name": "套餐A" },
        { "id": "b", "name": "套餐B" }
      ]
    }
  ]
}
```

特点很明确：

* 不关心库存
* 不关心价格
* 不关心组合是否存在

它只回答一个问题：

> **页面上有哪些规格可以被展示。**

---

### 2. SKU 列表：真正的商品实例

SKU 才是真正参与业务计算的对象：

```json
{
  "skuList": [
    {
      "skuId": "sku1",
      "price": 99,
      "stock": 10,
      "specs": {
        "color": "red",
        "size": "m",
        "package": "a"
      }
    },
    {
      "skuId": "sku2",
      "price": 109,
      "stock": 0,
      "specs": {
        "color": "red",
        "size": "l",
        "package": "a"
      }
    }
  ]
}
```

SKU 的本质是：

> **一组 specId → valueId 的映射 + 库存 / 价格等业务属性。**

它不关心展示顺序，也不存在嵌套关系。

---

## 四、标准 SKU 模型是如何工作的

### 1. 页面初始化

* 页面根据 specs 渲染所有规格
* 不裁剪、不隐藏

---

### 2. 用户选择某个规格值

例如：

```
color = red
```

系统并不是“进入下一层”，而是：

* 构造一个假设选择
* 用它去遍历 skuList

---

### 3. 为每个规格值计算状态

每一个规格值都会被判断为：

* ENABLED：存在且有库存
* OUT_OF_STOCK：存在但库存为 0
* DISABLED：不存在该组合
* SELECTED：当前选中

> **UI 只展示状态，不参与判断。**

---

## 五、新旧方案的本质差异

| 维度     | 数组套数组 | 标准 SKU 模型 |
| ------ | ----- | --------- |
| 数据结构   | 嵌套    | 扁平        |
| 规格扩展   | 困难    | 简单        |
| 状态表达   | 隐式    | 显式        |
| UI 复杂度 | 高     | 低         |
| 业务集中度  | 低     | 高         |

---

## 六、总结

数组套数组，本质上是在描述：

> **“用户的选择路径”**

而标准 SKU 模型描述的是：

> **“规格组合与业务状态”**

真正可维护的 SKU 设计，一定是：

* 规格只负责展示
* SKU 只负责业务
* 状态由统一规则计算

当你完成这个认知转变后，

> **SKU 就不再是一个 UI 难题，而是一个清晰的状态建模问题。**
