# Android SKU Demo

[运行效果](images/sku.gif)  
**从数组套数组到标准 SKU 模型的规格选择实现**

> 一个用于理解和实现 **电商 SKU 规格选择** 的 Android Demo 项目
> 聚焦 **数据模型设计 + 状态计算思路**，而不是 UI 技巧

---

## 这个项目解决什么问题？

在电商类 App 中，「SKU 规格选择」几乎是必做功能，但它也是**最容易写乱的模块之一**：

* 点一个规格，其他规格要跟着变灰 / 可选
* 有些组合存在但没库存
* 有些组合根本不存在
* 规格一多，逻辑就开始写进 Adapter / Fragment

很多项目的第一次实现，都会走向一种结构：

> **数组套数组（嵌套规格树）**

而这个 Demo，正是围绕一个问题展开：

> **SKU 规格，真的应该用“嵌套结构”来建模吗？**

---

## 常见但有问题的实现方式：数组套数组

以「颜色 / 尺码 / 套餐」为例，很多实现会按“选择路径”来组织数据：

```
红色
 ├── M
 │    ├── 套餐A
 │    └── 套餐B
 └── L
      ├── 套餐A
蓝色
 └── M
      └── 套餐A
```

这种方式在一开始看起来很直观，但很快会暴露问题：

* ❌ **规格顺序被写死**（颜色 → 尺码 → 套餐）
* ❌ **扩展性极差**（加一个规格基本等于重构）
* ❌ **无法清晰区分**

    * 不存在的组合
    * 存在但无库存的组合
* ❌ **业务逻辑被迫写进 UI**

本质原因只有一个：

> **规格和 SKU 被混在了一起建模**

---

## 这个项目采用的核心思路：标准 SKU 模型

### 一句话版本

> **规格只负责展示，SKU 才是商品本体**

---

### 1️⃣ 规格列表（Spec List）

规格用于渲染 UI，本身**不包含库存和价格信息**：

```json
{
  "specs": [
    {
      "specId": "color",
      "name": "颜色",
      "values": [
        { "id": "red", "name": "红色" },
        { "id": "blue", "name": "蓝色" }
      ]
    },
    {
      "specId": "size",
      "name": "尺码",
      "values": [
        { "id": "m", "name": "M" },
        { "id": "l", "name": "L" }
      ]
    }
  ]
}
```

特点：

* 规格是**扁平的**
* 规格数量、顺序不受限制
* 只用于“页面展示”

---

### 2️⃣ SKU 列表（Sku List）

SKU 才是真正的商品实例：

```json
{
  "skuId": "sku1",
  "price": 99,
  "stock": 10,
  "specs": {
    "color": "red",
    "size": "m"
  }
}
```

SKU 的本质是：

> **一组 `specId → valueId` 的映射**

这使得：

* 任意规格组合都能表达
* 不依赖规格顺序
* 判断逻辑可以完全统一

---

## 核心设计：SkuEngine

这个 Demo 的核心并不是 UI，而是一个 **纯业务组件：SkuEngine**

### SkuEngine 负责什么？

* 管理当前已选规格
* 计算每个规格值的状态：

    * 可选（ENABLED）
    * 不存在（DISABLED）
    * 存在但无库存（OUT_OF_STOCK）
    * 已选中（SELECTED）
* 匹配当前选中的 SKU
* 支持默认选中规则

👉 **所有 SKU 判断逻辑，都集中在 Engine 内部**

---

### 核心思想：假设选择 + 匹配

对于页面上的**每一个规格值**，Engine 都会做一次“假设选择”：

1. 将该值加入当前已选规格
2. 遍历所有 SKU
3. 判断：

    * 是否存在匹配的 SKU
    * 是否有库存

UI 层只拿到结果状态，不参与任何业务判断。

---

## 项目架构一览

```
UI (Fragment / Adapter)
        ↓
ViewModel（状态中转）
        ↓
SkuEngine（规格 & SKU 计算）
        ↓
Repository（本地 JSON / 数据源）
```

职责划分非常清晰：

| 层级         | 职责              |
| ---------- | --------------- |
| Repository | 提供规格 & SKU 原始数据 |
| SkuEngine  | 纯业务计算           |
| ViewModel  | UI 状态管理         |
| UI         | 只负责展示           |

## 为什么这个 Demo 适合参考？

* ✅ 不依赖具体 UI 形态
* ✅ 规格数量可扩展
* ✅ 状态逻辑集中、可测试
* ✅ 适用于商品 / 票务 / 套餐等场景

这不是一个“炫技巧”的 Demo，而是一个**可长期复用的 SKU 设计范式**。

## 结语

SKU 规格选择的复杂性，**不在 UI，而在建模方式**。

当你把：

* 规格 → 展示
* SKU → 组合
* 状态 → 计算结果

这三件事分清楚，SKU 问题会变得异常清晰。
